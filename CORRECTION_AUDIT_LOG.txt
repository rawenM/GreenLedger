┌───────────────────────────────────────────────────────────────┐
│  ⚠️  PROBLÈME RENCONTRÉ                                       │
└───────────────────────────────────────────────────────────────┘

Erreur: #1452 - Cannot add or update a child row: a foreign key 
constraint fails

Cause: Les données de test référencent des utilisateurs (ID 1, 2, 3)
qui n'existent pas dans votre table `user`.


┌───────────────────────────────────────────────────────────────┐
│  ✅ SOLUTION 1: UTILISER LE FICHIER CORRIGÉ                   │
└───────────────────────────────────────────────────────────────┘

J'ai corrigé le fichier database_audit_log.sql

Dans MySQL Workbench:

1. File → Open SQL Script
2. Sélectionner: database_audit_log.sql (le fichier corrigé)
3. Exécuter (icône éclair ⚡)

Les données de test utilisent maintenant NULL pour user_id,
ce qui est autorisé.


┌───────────────────────────────────────────────────────────────┐
│  ✅ SOLUTION 2: VERSION SIMPLE (SANS DONNÉES DE TEST)         │
└───────────────────────────────────────────────────────────────┘

Utilisez: database_audit_log_simple.sql

Ce fichier crée UNIQUEMENT la table, sans données de test.

Dans MySQL Workbench:

1. File → Open SQL Script
2. Sélectionner: database_audit_log_simple.sql
3. Exécuter (icône éclair ⚡)

La table sera créée vide, prête à recevoir les vrais logs.


┌───────────────────────────────────────────────────────────────┐
│  ✅ SOLUTION 3: SUPPRIMER ET RECRÉER                          │
└───────────────────────────────────────────────────────────────┘

Si la table existe déjà avec l'erreur, supprimez-la d'abord:

Dans MySQL Workbench, exécutez:

DROP TABLE IF EXISTS audit_log;

Puis exécutez un des fichiers SQL ci-dessus.


┌───────────────────────────────────────────────────────────────┐
│  🔍 VÉRIFIER QUE ÇA A FONCTIONNÉ                              │
└───────────────────────────────────────────────────────────────┘

Exécutez dans MySQL Workbench:

SELECT COUNT(*) FROM greenledger.audit_log;

Résultat attendu:
• Avec database_audit_log.sql: 5 (les 5 logs de test)
• Avec database_audit_log_simple.sql: 0 (table vide)

Les deux sont corrects! ✅


┌───────────────────────────────────────────────────────────────┐
│  📊 VOIR LES LOGS DE TEST (SI VOUS AVEZ UTILISÉ LE FICHIER 1) │
└───────────────────────────────────────────────────────────────┘

SELECT * FROM greenledger.audit_log ORDER BY created_at DESC;

Vous devriez voir:
• 1 connexion admin (il y a 5 minutes)
• 1 tentative échouée (il y a 4 minutes)
• 1 connexion réussie (il y a 3 minutes)
• 1 validation de compte (il y a 2 minutes)
• 1 fraude détectée (il y a 1 minute)


┌───────────────────────────────────────────────────────────────┐
│  💡 POURQUOI user_id PEUT ÊTRE NULL?                          │
└───────────────────────────────────────────────────────────────┘

C'est normal et voulu! user_id peut être NULL pour:

• Tentatives de connexion échouées (utilisateur pas encore connecté)
• Inscriptions (utilisateur pas encore créé)
• Actions système automatiques
• Logs de test

Quand l'utilisateur est connecté, user_id sera rempli automatiquement
par le service AuditLogService.
